Wowhead Tooltips

A comprehensive guide and FAQ for using the power.js script (https://wow.zamimg.com/widgets/power.js) to display Wowhead tooltips.

1. Introduction

Welcome to the complete guide for Wowhead Tooltips, powered by the power.js widget (https://wow.zamimg.com/widgets/power.js). The purpose of this document is to consolidate knowledge, clarify functionalities, and provide a detailed reference for developers and Wowhead staff working with or documenting this powerful tool.

The https://wow.zamimg.com/widgets/power.js script allows websites to display rich, interactive tooltips for various World of Warcraft (and other Blizzard games) entities by simply linking to Wowhead or using specific data attributes. This guide primarily focuses on its application for World of Warcraft.

This document synthesizes information from the analysis of the https://wow.zamimg.com/widgets/power.js script itself, observed API usage patterns, and common integration practices.

2. Core Components: power.js and whTooltips

The magic behind Wowhead Tooltips lies in the https://wow.zamimg.com/widgets/power.js script. To initialize and configure its behavior on a webpage, a global JavaScript object named whTooltips (historically, wowhead_tooltips was also used) must be defined before https://wow.zamimg.com/widgets/power.js is loaded.

Basic HTML Setup Example:

<script>
const whTooltips = {
  colorLinks: true,  // Allows Wowhead to apply quality colors to links
  iconizeLinks: true, // Allows Wowhead to add icons next to links
  renameLinks: true,  // Allows Wowhead to rename link text to the official entity name
  iconSize: 'tiny'   // Default size for icons ('tiny', 'small', 'medium', 'large')
};
</script>
<script src="https://wow.zamimg.com/widgets/power.js"></script>

Key Parameters of the whTooltips Object:

  • colorLinks: true — Allows https://wow.zamimg.com/widgets/power.js to apply standard Wowhead quality colors to links.
  • iconizeLinks: true — Allows https://wow.zamimg.com/widgets/power.js to automatically add icons next to links.
  • renameLinks: true — Allows https://wow.zamimg.com/widgets/power.js to change the link text to the official entity name from the Wowhead database.
  • iconSize: 'tiny' — Sets the default size for icons added by the script. Available values: 'tiny', 'small', 'medium', 'large'.

Important Note on whTooltips Configuration:

The whTooltips object is a global configuration for all tooltips on a single page. The https://wow.zamimg.com/widgets/power.js script reads this object once upon initialization. This means settings like iconSize, colorLinks, and renameLinks will apply universally to all Wowhead tooltips rendered by that instance of https://wow.zamimg.com/widgets/power.js on the page. Individual customization of these parameters for each separate link via whTooltips is not possible. Instead, to change the appearance of individual links (e.g., text or color), plugins or custom scripts should directly modify the HTML attributes of these links, after first setting the corresponding global flags in whTooltips (e.g., colorLinks: false, renameLinks: false) to prevent https://wow.zamimg.com/widgets/power.js from overriding these changes.

3. Understanding power.js Logic

To effectively use https://wow.zamimg.com/widgets/power.js, it's important to understand how it handles different game versions, languages, and data types.

3.1. Game Environments and Versions

The https://wow.zamimg.com/widgets/power.js script internally manages data for different games and their states (live, PTR, beta, classic, etc.) using a system of "data environments" and "data trees".

Internal Identifiers in power.js:

Analysis of the https://wow.zamimg.com/widgets/power.js script reveals the use of internal constants such as:

  • WH.dataEnv: Defines specific game versions and states (e.g., MAIN: 1 for Retail Live, CLASSIC: 4 for Classic Era, WRATH: 8 for WotLK Classic, etc.).
  • WH.dataEnvKey: Maps numerical IDs to text keys (e.g., 1:"live", 4:"classic", 8:"wrath").
  • WH.dataTree: Groups environments into broader categories (e.g., RETAIL: 1, CLASSIC: 4).

Mapping to the domain Parameter:

The domain parameter in links or data-wowhead attributes is used to tell https://wow.zamimg.com/widgets/power.js which game version and language to display. 3.2. Languages

https://wow.zamimg.com/widgets/power.js supports tooltip localization for multiple languages. The main language domains that can be used in the domain parameter include:

en - English (often default)
ru - Russian
de - German
es - Spanish
fr - French
it - Italian
pt - Portuguese (usually PT-BR)
ko - Korean
cn - Chinese (Simplified)
tw - Chinese (Traditional)

Internally, https://wow.zamimg.com/widgets/power.js maps these language codes to numerical locale IDs (e.g., via the WH.getLocaleFromDomain.L object, where ru:7, de:3, etc.).

3.3. Matching Language and Game Version

To correctly display tooltips for a specific game version in a specific language, the combined domain parameter is used.

  • Format language.version: This is the preferred and most correct format. Examples: ru.classic, de.wotlk, cn.cata, fr.mop-classic. The https://wow.zamimg.com/widgets/power.js script parses such a domain, applying the specified language locale to the data of the corresponding game version.
  • Format version.language (not recommended): Although some systems may attempt to automatically correct this format (e.g., classic.ru to ru.classic), the language.version format should always be used for reliability.
  • Version only (e.g., domain=classic, domain=wotlk): In this case, https://wow.zamimg.com/widgets/power.js will use data for the specified game version. The default language for such a version is usually English, unless otherwise determined by the specific implementation logic (e.g., a plugin).
  • Language only (e.g., domain=ru, domain=de): This implies the Retail version of the game in the specified language.

4. Wowhead Entities (Data Types)

Wowhead deals with numerous different types of game data, referred to as "entities." The https://wow.zamimg.com/widgets/power.js script uses an internal system of numerical identifiers for these types, defined in the WH.Types object. For better understanding, these entities can be conceptually grouped based on their role and prevalence.

4.1. Entity Classification

Below is an attempt to classify entities based on their typical use and importance.

Category 1: Core (Mandatory) Entities

These are fundamental types necessary for displaying key World of Warcraft game information. They are widely used and are often the objects of requests for tooltips.

  • item WH.Types.ITEM: 3
    Equipment, consumables, reagents, etc.
  • spell WH.Types.SPELL: 6
    Player/NPC abilities, item effects.
  • quest WH.Types.QUEST: 5
    In-game quests.
  • npc WH.Types.NPC: 1
    Any computer-controlled characters.
  • zone WH.Types.ZONE: 7
    Locations, dungeons, raids, cities.
  • achievement WH.Types.ACHIEVEMENT: 10
    In-game achievements.
  • object WH.Types.OBJECT: 2
    Interactive world objects (chests, ore nodes, herbs, etc.).
  • faction WH.Types.FACTION: 8
    In-game factions and reputations.
  • player-class (string ID: class) WH.Types.PLAYER_CLASS: 13
    Playable classes (Warrior, Mage, etc.).
  • race (string ID: race) WH.Types.RACE: 14
    Playable races (Human, Orc, etc.).
  • skill (string ID: skill) WH.Types.SKILL: 15
    Primary and secondary professions, weapon skills.

Category 2: Supplementary (Optional) Entities

These types usually extend or supplement core entities, or relate to specific game systems and mechanics (e.g., from particular expansions). They may not always be available or relevant.

  • item-set WH.Types.ITEM_SET: 4
    Equipment sets with set bonuses.
  • title WH.Types.TITLE: 11
    Character titles.
  • event WH.Types.EVENT: 12
    In-game holidays, seasonal and world events.
  • currency WH.Types.CURRENCY: 17
    Various types of in-game currency (Honor, Valor, specific event tokens).
  • affix WH.Types.AFFIX: 40
    Mythic+ dungeon modifiers.
  • transmog-set WH.Types.TRANSMOG_SET: 101
    Predefined appearance sets for transmogrification.
  • outfit WH.Types.OUTFIT: 110
    User-saved equipment sets for quick changes.
  • hunter-pet WH.Types.HUNTER_PET: 9
    Abilities and information about Hunter class pets.
  • guide WH.Types.GUIDE: 100
    Guide articles on the Wowhead site.
  • icon WH.Types.ICON: 29
    Individual game icons (rarely used for direct tooltips, more often as part of other entities).
  • sound WH.Types.SOUND: 19
    In-game sound files.
  • ... and many other more specific types, such as building, follower, mission, azerite-essence, azerite-essence-power, storyline, profession-trait, battle-pet-ability, trading-post-activity. Refer to the WH.Types object in the https://wow.zamimg.com/widgets/power.js script for a complete list.

Category 3: Standalone System Entities

These types often define basic game conditions or system elements and can exist independently. Some may overlap with Core entities as they are fundamental, but their context can be more general.

  • resource WH.Types.RESOURCE: 27
    E.g., garrison or order hall resources.
  • race WH.Types.RACE: 14
    As a general concept defining available races (also in Core).
  • player-class WH.Types.PLAYER_CLASS: 13
    As a general concept defining available classes (also in Core).
  • zone WH.Types.ZONE: 7
    As a general concept of a territory (also in Core).
  • faction WH.Types.FACTION: 8
    As a general concept of allegiance (also in Core).

Note on Internal WH.Types IDs: The https://wow.zamimg.com/widgets/power.js script uses numerical identifiers for types. Some correspond directly to string names, while others require mapping (e.g., PLAYER_CLASS in power.js for the string identifier class that might be used in URLs or attributes). Refer to the WH.Types object in the script for a full list of numerical IDs and their string representations (if available).

5. Working with XML Data from Wowhead

Wowhead provides XML feeds for many entities, allowing developers to retrieve structured data for further processing, custom display, or integration with other systems. This method is an alternative or supplement to using the https://wow.zamimg.com/widgets/power.js script.

5.1. Environment Requirements for XML Parsing

To successfully parse XML data from Wowhead on the server-side (e.g., using PHP), the following server environment is typically required:

  • Hosting: Any standard web hosting with PHP support.
  • PHP Version: PHP 7.4 or higher is recommended. Older versions (e.g., PHP 5.6, 7.0-7.3), while they might work, are outdated and may have different levels of support for XML extensions.
  • PHP Extensions:
    • libxml: A fundamental library for XML manipulation in PHP. Usually enabled by default in most hosting environments.
    • SimpleXML: A user-friendly extension for parsing XML documents. Also typically enabled by default. You can check its availability using the PHP function class_exists('SimpleXMLElement').
    • cURL: The preferred extension for making HTTP requests to external servers, like Wowhead, to fetch XML files. It offers flexible options for request configuration.
    • allow_url_fopen: If the cURL extension is not available or configured, the allow_url_fopen directive in the PHP configuration file (php.ini) must be set to On. This allows PHP functions like file_get_contents() to work with URLs for retrieving remote content. However, cURL is generally considered a more secure and versatile option.

5.2. Constructing URLs for XML Requests

To obtain XML data for a specific entity, append the parameter &xml=1 (or simply &xml) to the standard URL of that entity on the Wowhead site. The case of the xml parameter usually does not matter.

General URL Template for XML:

https://{wowhead_domain}/{optional_language_path}/{optional_game_version_path}/{entity_type}={entity_id}&xml=1

Example URLs for Fetching XML:

  • Item (Retail, Russian):
    https://www.wowhead.com/ru/item=19019&xml=1
  • Item (Retail, German):
    https://www.wowhead.com/de/item=19019&xml=1
  • Quest (Classic, English default):
    https://www.wowhead.com/classic/quest=1109&xml=1
  • Quest (Classic, Russian):
    https://www.wowhead.com/classic/ru/quest=1109&xml=1
  • Spell (Retail, Russian):
    https://www.wowhead.com/ru/spell=33786&xml=1
  • Item with Additional Parameters (e.g., gems):
    https://www.wowhead.com/ru/item=25697&gems=23121&xml=1
    (Parameters like gems, ench, pcs, etc., are added to the URL before the &xml=1 parameter.)

Complete URL lists for different game versions and languages can be constructed following this pattern by substituting the appropriate version and language segments.

5.3. Data Extractable from XML

The XML feed from Wowhead can provide a wide range of information about the requested entity. The XML structure may vary slightly for different entity types, but the following key data can usually be expected:

XML Tag (Example) Description of Data
<name>...</name> Official localized name of the entity. Often contains CDATA.
<icon displayId="...">...</icon> Icon filename (without extension, e.g., inv_sword_67) and sometimes its unique displayId.
<link>...</link> A "pretty" URL (slug) to the entity's page on the Wowhead site, often including the name.
<quality id="...">...</quality> Item quality (e.g., "Epic", "Rare") and its numerical identifier.
<class id="...">...</class> Item class (e.g., "Weapon", "Armor").
<subclass id="...">...</subclass> Item subclass (e.g., "Swords", "Mail").
<htmlTooltip>...</htmlTooltip> The full HTML code of the standard Wowhead tooltip. Can be useful for direct display if not using https://wow.zamimg.com/widgets/power.js. Often contains CDATA.
<json>...</json> General data about the object in JSON format (within CDATA). Requires additional JSON parsing.
<jsonEquip>...</jsonEquip> Equipment-specific data in JSON format (stats, armor/weapon type, requirements, etc., within CDATA).
Various specific tags Depending on the entity type, tags with quest level, requirements, spell school for spells, rewards, etc., may be present.
<error>...</error> If the requested object is not found or another error occurs on Wowhead's side, the XML may contain this tag with an error description.
To correctly parse data wrapped in CDATA (e.g., the content of <name> or <htmlTooltip>), appropriate methods of your XML parser should be used (e.g., casting to string in SimpleXML).

6. https://wow.zamimg.com/widgets/power.js vs. XML: Advantages and Limitations

When working with Wowhead data, developers have two main ways to retrieve and display it: using the official JavaScript widget power.js or by directly parsing XML feeds. Each approach has its strengths and weaknesses.

https://wow.zamimg.com/widgets/power.js

Advantages:

  • Ease of Integration: Simply include the script and define the global whTooltips object.
  • Automation: The script automatically finds links, generates tooltip HTML, adds icons, and applies quality colors.
  • Interactivity: Provides "live" tooltips with features like item level scaling, 3D model views, etc., within the tooltip itself.
  • Consistent Style: Ensures a consistent and recognizable Wowhead-style appearance for tooltips.
  • Data Freshness: Tooltip information is always up-to-date as it's loaded directly from Wowhead servers on display.

Limitations:

  • Global Configuration: Settings in whTooltips (e.g., iconSize, colorLinks) apply to all tooltips on the page, making individual link behavior customization difficult without workarounds.
  • External Resource Dependency: Tooltip functionality is entirely dependent on the availability of wow.zamimg.com servers.
  • Less Data Control: Developers receive pre-formatted HTML tooltips; direct access to raw data for custom processing is limited.
  • Potential Conflicts: Incorrect implementation or conflicts with other JavaScript libraries on the page can interfere with its operation.

XML Feeds

Advantages:

  • Direct Data Access: Retrieve structured raw data (names, stats, descriptions, icon IDs, etc.) for custom use.
  • Full Display Control: Ability to create a completely custom design for tooltips or any other information presentation.
  • Local Caching: XML responses can be efficiently cached on your server, reducing dependency on Wowhead's availability and significantly speeding up data loading for users.
  • Offline Work (with cache): If data is cached, information can be available even during temporary connectivity issues with Wowhead.
  • Specific Detail Retrieval: XML might contain more details or attributes than are displayed in a standard tooltip.

Limitations:

  • Implementation Complexity: Requires server-side logic (e.g., in PHP) to make HTTP requests, parse the XML structure, and then format the HTML.
  • No Built-in Interactivity: All interactivity similar to power.js tooltips must be implemented manually.
  • Parser Maintenance: Wowhead's XML feed structure could theoretically change, requiring updates to the parsing code.
  • Load on Wowhead Servers: Without proper and efficient caching, frequent requests to XML feeds can create excessive load on Wowhead's servers and potentially lead to access restrictions.
  • No Standard Icons/Styling "Out-of-the-Box": All visual presentation, including loading and displaying icons, is the developer's responsibility.

The choice between these two approaches (or a combination thereof) depends on the specific project requirements, desired level of customization, and available development resources.

7. Useful Tips & Recommendations

Below are some tips that can help when working with Wowhead Tooltips and debugging your projects.

Debugging https://wow.zamimg.com/widgets/power.js:

  • Always check your browser's JavaScript console (usually opened with F12, "Console" tab) for errors. Any JavaScript errors on the page can interfere with the proper functioning of the tooltip script.
  • Ensure the global JavaScript object whTooltips is defined on the page before the https://wow.zamimg.com/widgets/power.js script is included. You can verify its existence and content by typing whTooltips into the browser console after the page loads.
  • Verify that the https://wow.zamimg.com/widgets/power.js script itself is successfully loaded by the browser (check the "Network" tab in developer tools). The response status should be 200 OK.
  • If Wowhead links are not being processed (no icons or tooltips appear), ensure that the <a> HTML tags either have an href attribute pointing to the corresponding Wowhead page or contain a correctly formatted data-wowhead="type=id&domain=language.version&..." attribute.

Working with the domain Parameter:

  • Always use lowercase letters for domain parameter values (e.g., ru.classic, not RU.Classic).
  • For combined domains (language + game version), always use the format language.version (e.g., ru.classic, de.wotlk). The version.language format (e.g., classic.ru) is incorrect and may not work or may be processed unpredictably.
  • Remember that PTR and Beta versions usually only support English for tooltips, even if you specify a different language prefix in the domain parameter.

XML and Performance:

  • Always cache XML responses! If you are using XML feeds to retrieve data server-side, it's crucial to implement a caching mechanism. In WordPress, the Transients API is excellent for this.
    • Set a reasonable cache lifetime: e.g., 12-24 hours for relatively static data (item/spell names) and shorter (1-6 hours) for data that might change more frequently, or for 404 errors.
  • Avoid making "live" XML requests to Wowhead for every link on a page with every page load. This will create a huge load and slow down your site. It's better to fetch data on the first access to an object and then store it in the cache.

Customizing Appearance (when using power.js):

  • Although https://wow.zamimg.com/widgets/power.js styles the tooltips itself, you can override some of its CSS styles using more specific selectors or the !important directive in your theme's CSS. However, do this cautiously, as it might break the tooltip's layout or stop working after updates to the power.js script.
  • To change the appearance of the links themselves (text, color), use appropriate shortcode attributes (like name, color, style in the plugin discussed earlier) or directly modify the HTML attributes of the links, after ensuring that renameLinks: false and colorLinks: false are set in whTooltips.

"Tricks" and Advanced Aspects:

  • Extended Parameters in data-wowhead: The data-wowhead attribute can contain not only basic type=id&domain=language.version, but also other parameters that https://wow.zamimg.com/widgets/power.js might consider for a more accurate tooltip display. For items, these can include:
    • lvl=X (character level for stat scaling)
    • gems=ID1:ID2... (inserted gems)
    • ench=ID (enchantment)
    • pcs=ID1:ID2... (set pieces for displaying set bonuses)
    • bonus=ID1:ID2... (bonus IDs for items with variants)
    These parameters allow the script to display the tooltip as if the item is configured in a specific way.
  • Using Wowhead's Quality CSS Classes: If the colorLinks: true option is set (or not set and defaults to true), the https://wow.zamimg.com/widgets/power.js script adds CSS classes to links based on item quality (e.g., q0, q1, q2, q3, q4, q5, q6, q7 for artifacts and legendaries). You can use these classes in your CSS for additional styling if Wowhead's default colors don't suit you, but you want to maintain color differentiation by quality.
  • Checking Data Environment Activity: The https://wow.zamimg.com/widgets/power.js script has internal functions (e.g., WH.isDataEnvActive(envId)) that could theoretically be used to check if a particular data environment (like PTR, Beta) is active on Wowhead's side. This might be useful for dynamically displaying links or information on a site.
Please remember that internal functions and the structure of https://wow.zamimg.com/widgets/power.js can be changed by Wowhead without prior notice. Directly using undocumented internal functions is not recommended for production sites due to the fragility of such an approach.
